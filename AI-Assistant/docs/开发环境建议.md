# 开发环境建议

本文档用于说明当前项目在改造阶段推荐采用的开发环境方案，重点回答以下问题：

1. 哪些服务适合跑在本机？
2. 哪些基础组件适合放到 Docker 中？
3. 端口如何规划？
4. 当前阶段为什么不建议一开始就全量容器化？
5. 后续项目进入正式部署阶段时，可以如何演进？

本文档以当前项目状态为前提：

- Python RAG 服务正在从开源项目改造为可嵌入 Java 企业知识库的独立能力服务
- Redis 已经运行在 Docker 中
- 当前前端界面可以先保留，不作为改造重点
- Embedding 初期采用 Ollama `qwen3-embedding:4b`
- 后续再引入 `BGE-M3` 做对比评测

---

## 1. 推荐的整体环境方案

当前阶段最推荐的开发环境方案是：

**本机开发 + Docker 承载基础组件**

建议划分如下：

### 本机运行

- Python RAG 服务
- Ollama
- Java 主项目
- ChromaDB 持久化目录

### Docker 运行

- Redis
- MinIO（后续接入时）
- MySQL（如果后续仍需要）

可以概括为一句话：

**Python、Java、Ollama 跑本机，Redis 和后续的 MinIO 跑 Docker，Chroma 先走本地目录持久化。**

---

## 2. 为什么推荐这种方式

### 2.1 最适合当前改造阶段

当前项目还处于高频修改阶段，后端接口、配置结构、文档处理链路、向量检索逻辑都可能持续变化。

如果此时把 Python 服务也放进 Docker，那么每次改代码后都要频繁：

- 重建镜像
- 重启容器
- 排查容器内依赖
- 查看容器日志

这会显著降低开发效率。

相反，本机直接运行 Python 服务更适合当前阶段：

- 改完代码即可重启或热更新
- 更容易单步调试
- 更容易查看日志和本地文件
- 更适合和 VSCode 中的 agent 协同开发

### 2.2 与当前 Redis 方案兼容

你现在已经把 Redis 放在 Docker 里了，这个方案本身就是合理的。

Redis 这类基础服务有几个特点：

- 相对稳定，不会频繁改代码
- 容器化部署简单
- 很适合独立成基础组件

因此 Redis 继续放 Docker 是推荐做法。

### 2.3 更方便调试文档处理链路

当前改造重点之一是文档处理链路，包括：

- 文件解析
- 文本切块
- 向量化
- 写入 ChromaDB
- 查询召回

这些环节都非常依赖调试体验，尤其你经常需要看：

- 原始上传文件
- 解析后的文本内容
- chunk 切割效果
- metadata 是否正确
- 向量检索结果是否合理

如果这些都在本机环境完成，定位问题会更快。

### 2.4 便于前期继续使用 Ollama

你当前初期方案是使用 Ollama `qwen3-embedding:4b`，那么最自然的做法就是让 Ollama 在本机直接运行。

这样 Python 服务可以直接访问：

```text
http://localhost:11434
```

无需额外处理容器网络问题。

## 3. 当前阶段推荐的组件部署方式

### 3.1 Python RAG 服务

推荐方式：

- 本机运行
- 使用 VSCode 直接调试
- 使用本地虚拟环境

推荐原因：

- 开发效率最高
- 方便改接口和调逻辑
- 容易查看日志、文件和向量数据

### 3.2 Java 主项目

推荐方式：

- 本机运行

推荐原因：

- 当前重点是 Python 与 Java 的接口联调
- 本机跑 Java 更便于同时看前后端日志

### 3.3 Ollama

推荐方式：

- 本机运行

推荐原因：

- 当前 Embedding 方案初期基于 Ollama
- 本机运行最简单
- 与 Python 本机调试链路最短

### 3.4 Redis

推荐方式：

- Docker 运行

推荐原因：

- 稳定基础组件，适合容器化
- 当前你已经这样做了
- 后续 Celery 或缓存升级都可以直接复用

### 3.5 MinIO

推荐方式：

- Docker 运行

推荐原因：

- 它本质上就是独立对象存储服务
- 和 Python/Java 的关系清晰
- 很适合容器化

当前阶段如果还没开始接 MinIO，可以暂时不装，等文件入库接口开始改造时再加。

### 3.6 ChromaDB

推荐方式：

- 当前阶段不单独容器化
- 先作为本地持久化目录使用

推荐原因：

- 当前项目代码本身就是通过本地路径做持久化
- 更方便调试 collection、metadata 和删除逻辑
- 不需要额外引入容器管理复杂度

对应当前配置项：

- `CHROMA_PERSIST_DIR=./chroma_db`

## 4. 推荐的端口规划

下面是一份适合当前项目阶段的推荐端口规划。

| 组件 | 推荐端口 | 说明 |
|---|---:|---|
| Java 主项目 | 8080 / 8081 | 视你的 Java 项目而定 |
| Python RAG 服务 | 8000 | FastAPI 默认可直接沿用 |
| Ollama | 11434 | 默认端口 |
| Redis | 6379 | Docker 映射标准端口 |
| MinIO API | 9000 | 对象存储接口 |
| MinIO Console | 9001 | 管理控制台 |

如果以后本机端口冲突，可以统一调整，但建议尽量保持这一套默认值，便于文档统一。

## 5. 推荐的目录与数据落地方式

### 5.1 Python 项目目录

当前阶段建议继续围绕 `rag/backend` 作为 Python RAG 服务根目录改造。

### 5.2 Chroma 持久化目录

推荐使用固定目录，例如：

```text
rag/backend/chroma_db
```

或者后续更规范一些：

```text
rag/backend/data/chroma_db
```

### 5.3 上传文件目录

如果还保留本地上传链路，建议统一使用：

```text
rag/backend/uploads
```

后续如果切换到 MinIO，这部分本地上传目录可以逐步弱化。

### 5.4 日志目录

建议统一保留：

```text
rag/backend/logs
```

这样方便调试：

- 文档解析日志
- 检索日志
- 接口错误日志

## 6. 推荐的 Python 版本

当前阶段建议固定使用：

```text
Python 3.12.x
```

推荐原因：

1. 你前面的改造规划已经围绕 3.12 展开。
2. 当前依赖生态在 3.12 上更稳定。
3. 可以避免过新版本带来的依赖兼容问题。

建议避免在当前项目中混用多个 Python 主版本，尤其不要让环境漂移到 3.14。

## 7. 当前阶段建议的配置项

虽然具体值需要你自己根据本机环境填写，但建议尽早把下面这些配置项规划清楚。

### 7.1 Python 服务基础配置

- `HOST`
- `PORT`
- `DEBUG`
- `APP_NAME`
- `APP_VERSION`

### 7.2 Ollama 配置

- `OLLAMA_BASE_URL`
- `OLLAMA_LLM_MODEL`
- `OLLAMA_EMBEDDING_MODEL`

建议初期类似：

```env
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_EMBEDDING_MODEL=qwen3-embedding:4b
```

### 7.3 Chroma 配置

- `CHROMA_PERSIST_DIR`

建议初期类似：

```env
CHROMA_PERSIST_DIR=./chroma_db
```

### 7.4 Redis 配置

- `REDIS_HOST`
- `REDIS_PORT`
- `REDIS_DB`

如果 Redis 在 Docker 里并映射到宿主机，通常可以写成：

```env
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_DB=0
```

### 7.5 MinIO 配置

后续需要接入时建议预留：

- `MINIO_ENDPOINT`
- `MINIO_ACCESS_KEY`
- `MINIO_SECRET_KEY`
- `MINIO_SECURE`
- `MINIO_BUCKET`

### 7.6 API 鉴权配置

后续建议增加：

- `API_KEY`

用于 Java 服务调用 Python 服务时的接口鉴权。

### 7.7 DeepSeek 配置

如果后面切换或新增 DeepSeek 作为 LLM，建议预留：

- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `DEEPSEEK_MODEL`

## 8. 为什么当前不建议一开始就全量 Docker 化

当前阶段不建议直接进入下面这种模式：

- Python in Docker
- Java in Docker
- Ollama in Docker
- Chroma in Docker
- Redis in Docker
- MinIO in Docker

原因不是这种方式不行，而是它更像“部署环境”，不适合你当前的“高频改造和调试环境”。

当前如果全量容器化，通常会增加这些成本：

1. 代码改动后的验证成本变高。
2. 配置和挂载路径更复杂。
3. 日志查看和文件排错更麻烦。
4. 容器网络问题会干扰本应专注的业务改造。

因此，更合理的顺序应该是：

1. 先在本机 + Docker 基础组件环境下完成改造和联调。
2. 等接口稳定、链路跑通后，再整理 Dockerfile 和部署方案。

## 9. 推荐的阶段性演进路线

### 第一阶段：改造与联调阶段

建议环境：

- Python 本机
- Java 本机
- Ollama 本机
- Redis Docker
- Chroma 本地目录

目标：

- 跑通入库
- 跑通问答
- 跑通删除
- 跑通基础联调

### 第二阶段：能力增强阶段

建议新增：

- MinIO Docker
- Redis 继续复用
- 如有需要再引入 Celery

目标：

- 改造为 `file_url` 入库
- 增加异步任务能力
- 增加状态查询
- 做 Embedding 对比实验

### 第三阶段：部署准备阶段

建议再考虑：

- Python 服务 Docker 化
- 统一 `.env` 与 Docker Compose
- 固化日志、数据目录、端口映射

目标：

- 为测试环境或正式环境做准备

## 10. 当前阶段的推荐结论

结合当前项目目标，最适合你的开发环境方案是：

**Python RAG 服务、Java 主项目、Ollama 运行在本机；Redis 和后续 MinIO 运行在 Docker 中；ChromaDB 先使用本地目录持久化。**

这个方案最适合当前改造阶段的原因是：

1. 开发效率高
2. 调试链路短
3. 和现有 Redis 方案兼容
4. 适合前期基于 Ollama 快速验证
5. 后续也容易平滑演进到更正式的部署方式

如果只看一句话总结，可以写成：

**当前阶段优先采用“本机开发 + Docker 基础组件”的混合环境，而不是一开始就全量容器化。**
