# 文档切割流程

本文档基于当前项目代码，整理文档从上传、解析、切块到入向量库的完整流程，并说明当前切割粒度、切割策略，以及如何规避语义被切断的问题。

## 1. 整体流程概览

当前项目中的文档处理主链路如下：

1. 用户上传文件或从本地路径导入文件。
2. 系统将原始文件保存到本地上传目录。
3. 在 MySQL 的 `documents` 表中写入文档元数据记录。
4. 后台调用解析器，将原始文件解析成一个或多个 LangChain `Document` 对象。
5. 调用文本切割器，将这些 `Document` 再切成更小的 chunk。
6. 为每个 chunk 补充元数据。
7. 将 chunk 写入 ChromaDB，供后续检索问答使用。
8. 将切块数量、字符数、解析元数据回写到 MySQL。

主流程对应代码：

- 文档解析服务：[rag/backend/app/services/parser.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/services/parser.py)

## 5. 当前项目的文档切割策略

当前项目使用的是 `RecursiveCharacterTextSplitter`，不是简单按固定长度生硬截断。

对应代码：

- [rag/backend/app/services/vector_store.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/services/vector_store.py:183)

当前分隔符优先级如下：

```python
["\n\n", "\n", "。", "！", "？", "；", ".", "!", "?", ";", " ", ""]
```

对应代码：

- [rag/backend/app/services/vector_store.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/services/vector_store.py:186)

这意味着切割策略是递归式的：

1. 优先按段落空行切。
2. 如果一段仍然太长，再按普通换行切。
3. 如果还太长，再按中文句号、问号、感叹号、分号切。
4. 再按英文标点切。
5. 再按空格切。
6. 最后实在无法切开时，才退化到字符级切分。

因此，它本质上是：

- 优先按段落结构切
- 其次按句子边界切
- 最后才按更细粒度切

## 6. 当前项目怎么规避“语义被切割掉”

当前项目主要通过两种方式降低语义切断风险。

### 6.1 利用结构化分隔符优先切割

项目不会直接从第 1000 个字符硬切，而是优先找段落、换行、句号等自然边界。

对应代码：

- [rag/backend/app/services/vector_store.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/services/vector_store.py:183)
- [rag/backend/app/services/vector_store.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/services/vector_store.py:186)

这样做的意义是：

- 尽量让一个完整段落留在同一块里
- 尽量让一句话在同一块里结束
- 降低“上半句在 chunk A，下半句在 chunk B”这种情况

### 6.2 使用 overlap 保留上下文

当前项目为相邻 chunk 设计了 200 字符重叠区。

对应代码：

- [rag/backend/app/core/config.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/core/config.py:43)
- [rag/backend/app/services/vector_store.py](D:/ResumeProjects/AI-Assistant/rag/backend/app/services/vector_store.py:185)

这样做的作用是：

- 如果某个语义单元刚好跨 chunk 边界，前后两块都能保留一部分上下文
- 即使切割发生在不理想位置，也不会完全丢掉前后关联

可以理解为：

- `chunk_size` 决定单块最大体积
- `chunk_overlap` 决定相邻块之间的上下文冗余

## 8. 当前流程的优点

结合当前项目代码，现有流程的优点主要有：

1. 原始文件、业务元数据、向量数据三层分离，职责比较清晰。
2. PDF 按页解析，便于后续做来源定位。
3. Excel 按 Sheet 解析，适合表格型资料。
4. 切块使用递归分隔符，不是简单暴力截断。
5. 使用 overlap 缓解上下文断裂问题。
6. chunk 中保留 `doc_id`、`kb_id`、`filename` 等元数据，便于删除和回溯来源。

---

## 9. 当前流程的局限

当前实现已经具备基础可用性，但对企业文档场景来说，仍然属于基础版切块。

主要局限有：

1. Word、TXT 等通常是整篇先合成一个 `Document`，没有利用标题层级、章节结构。
2. 当前切块策略是“结构字符递归切分 + overlap”，还不是真正的语义切块。
3. chunk 元数据偏少，暂时没有部门、权限、版本、标题等更强业务字段。
4. 当前是按 `kb_id` 拆 collection，后续如果要支持更复杂的统一检索和权限过滤，可能需要重新设计。

## 10. 当前项目的完整切割链路总结

可以把当前项目的文档切割流程概括为：

```text
上传文件 / 本地导入
    ↓
保存原始文件到 uploads/kb_{kb_id}/
    ↓
写入 documents 表
    ↓
parser.parse() 按文件类型解析成一个或多个 Document
    ↓
vector_store._text_splitter() 使用 RecursiveCharacterTextSplitter 切块
    ↓
为 chunk 补充 doc_id / kb_id / filename / chunk_index 等元数据
    ↓
写入 ChromaDB 的 kb_{kb_id} collection
    ↓
回写 chunk_count / char_count / meta_info 到 MySQL
```

如果只看“切割”本身，那么当前项目的核心可以总结为一句话：

**先按文件结构做粗粒度解析，再按 1000 字符大小、200 字符重叠，用段落和句子优先的递归切块策略写入 ChromaDB。**
