# Embedding 选型对比

本文档用于记录当前项目在 Embedding 模型上的选型思路、阶段性决策、后续对比实验方案，以及未来补充数据后的完善方向。

当前阶段的目标不是一次性确定“绝对最优模型”，而是先完成项目落地，再逐步引入对比实验，用数据支撑后续正式选型。

---

## 1. 项目背景

当前项目的目标是将 `rag` 目录中的 Python RAG 能力，改造并嵌入到 Java 企业文档知识库项目中。

该场景下，Embedding 模型主要负责以下任务：

1. 将文档切块后的文本向量化。
2. 将用户问题向量化。
3. 在 ChromaDB 中完成向量检索。
4. 为后续问答链路提供候选上下文。

因此，Embedding 模型的选型会直接影响：

- 文档召回准确率
- 长文档切块后的匹配效果
- 中文企业文档场景下的稳定性
- 入库速度
- 查询速度
- 本地部署复杂度

---

## 2. 当前候选方案

结合当前项目阶段，候选方案暂定为两类：

### 2.1 方案 A：Ollama `qwen3-embedding:4b`

定位：

- 项目初期方案
- 用于快速打通本地 RAG 链路
- 与现有 Ollama 生态集成简单

特点：

- 本地部署直接
- 与当前项目已有的 Ollama 技术路线一致
- 便于快速验证“上传文档 -> 切块 -> 向量化 -> 检索 -> 问答”闭环
- 适合作为开发阶段和演示阶段的默认方案

当前决策：

- 项目初期优先采用 `qwen3-embedding:4b`

### 2.2 方案 B：`BGE-M3`（via `langchain-huggingface`）

定位：

- 项目正式优化阶段的候选方案
- 用于在企业知识库场景下进一步提升检索效果

特点：

- 更适合做正式的检索效果对比
- 对中文和多语言场景更有代表性
- 后续更方便扩展到 rerank、混合检索、多路召回等增强方案

当前决策：

- 在项目中后期引入 `BGE-M3`，与 `qwen3-embedding:4b` 做 A/B 对比实验

---

## 3. 为什么初期先选 Ollama `qwen3-embedding:4b`

当前项目仍处于改造阶段，重点是先把整个 RAG 服务能力从开源项目中抽离出来，并完成与 Java 企业知识库的对接。

在这个阶段，优先考虑的是：

1. 尽快跑通完整链路。
2. 降低依赖复杂度。
3. 减少环境配置成本。
4. 先验证架构是否成立，而不是一开始就把效果优化做到极致。

基于这个目标，`qwen3-embedding:4b` 更适合作为初期默认方案，原因如下：

### 3.1 与当前项目原有技术路线更接近

当前 `rag` 项目原本已经使用了 Ollama 体系：

- LLM：Ollama + Qwen
- Embedding：Ollama Embeddings

因此，继续沿用 Ollama 路线，可以用最小改动快速完成第一阶段验证。

### 3.2 部署和接入成本更低

相比引入 `sentence-transformers`、`torch`、`huggingface` 体系，Ollama 路线更轻量，初期更适合快速开发和调试。

### 3.3 更适合作为基线方案

如果后续要体现自己做过技术选型过程，那么必须先有一个“基线方案”。

`qwen3-embedding:4b` 非常适合作为这个 baseline：

- 容易接入
- 容易运行
- 容易完成初步演示

后续引入 `BGE-M3`，就可以形成清晰的“从可用，到更优”的技术演进路径。

---

## 4. 为什么后续要引入 `BGE-M3` 做对比

虽然 `qwen3-embedding:4b` 适合作为项目初期方案，但如果项目定位是企业文档知识库，那么后续仍然有必要引入更正式的对比方案。

引入 `BGE-M3` 的主要目的不是推翻前面的方案，而是用来验证：

1. 检索准确率是否能进一步提升。
2. 中文长文档场景下是否更稳定。
3. 对复杂企业文档的 chunk 匹配是否更好。
4. 是否值得为效果提升接受更高一点的部署复杂度。

这一步的价值在于：让项目从“能跑”升级为“做过效果验证和工程选型”。

---

## 5. 当前阶段的选型结论

现阶段的阶段性结论如下：

1. 项目初期先采用 Ollama `qwen3-embedding:4b`，优先完成本地 RAG 闭环验证。
2. 在完成 Java 知识库与 Python RAG 服务的基本对接后，再引入 `BGE-M3` 做正式对比。
3. 最终正式选型不靠主观判断，而以实验数据为依据。

可以概括为一句话：

**先用 `qwen3-embedding:4b` 跑通链路，再用 `BGE-M3` 做数据对比，最后基于效果和成本综合决策。**

## 6. 后续实验设计

为了体现项目中确实做过 Embedding 选型，而不是拍脑袋决定，后续需要补一轮正式的对比实验。

### 6.1 实验目标

对比以下两个方案在本项目场景中的表现：

- 方案 A：Ollama `qwen3-embedding:4b`
- 方案 B：`BGE-M3`

目标是评估它们在企业文档知识库场景中的：

- 检索效果
- 性能开销
- 部署成本
- 工程适配性

### 6.2 对比前提

为了保证实验公平，除 Embedding 模型外，其他条件尽量保持一致：

- 同一批测试文档
- 同一套切块策略
- 同一个 ChromaDB
- 同样的 `top_k`
- 同样的问答链路
- 同样的 Prompt 模板

### 6.3 建议测试数据集

建议从项目实际文档中挑选一批具有代表性的资料，例如：

- 企业制度文档
- 技术规范文档
- 项目方案文档
- 含表格的汇报材料
- 较长 PDF 文档

建议人工整理一批测试问题，规模可以先从 30 到 50 个开始。

每个问题建议标注：

- 问题文本
- 正确答案应来源于哪份文档
- 正确答案大致位于哪一页、哪一段、哪一类 chunk

这样后续才能做出相对可信的召回效果评估。

---

## 7. 建议对比指标

后续实验建议从以下几个维度打分。

### 7.1 检索效果指标

建议优先看：

- `Recall@5`
- `Recall@10`
- `MRR@10`

如果项目时间有限，至少保留：

- `Recall@5`
- 命中正确来源的比例

### 7.2 问答效果指标

虽然 Embedding 不直接生成答案，但它会影响最终问答质量，因此建议补充观察：

- 返回的来源文档是否正确
- 返回的来源片段是否足以支撑答案
- 最终答案是否偏离原文

### 7.3 性能指标

建议记录：

- 单篇文档入库耗时
- 批量文档入库总耗时
- 单次查询向量检索耗时
- 首次模型加载耗时

### 7.4 资源与工程成本指标

建议记录：

- 模型体积
- 内存占用
- CPU/GPU 依赖情况
- 部署复杂度
- 与现有系统集成复杂度

## 9. 后续可补充的结论结构

当有了正式实验数据后，可以按下面的结构补齐最终结论。

### 9.1 如果 `BGE-M3` 效果更好

可以写成：

> 在相同文档集、相同切块策略、相同向量库配置下，`BGE-M3` 在 `Recall@5`、`MRR@10` 等检索指标上优于 Ollama `qwen3-embedding:4b`。虽然部署复杂度略高，但在企业知识库正式场景中，检索质量收益更明显，因此最终正式方案采用 `BGE-M3`。

### 9.2 如果两者效果接近

可以写成：

> 在当前项目文档规模和问题集下，`BGE-M3` 与 Ollama `qwen3-embedding:4b` 的检索效果差异不显著。考虑到 Ollama 路线部署更简单、维护成本更低，最终项目优先采用 Ollama `qwen3-embedding:4b` 作为正式方案。

### 9.3 如果 Ollama 方案更适合当前项目

可以写成：

> 虽然 `BGE-M3` 在通用检索任务中具有较强能力，但在本项目当前文档规模、硬件资源和部署条件下，Ollama `qwen3-embedding:4b` 在效果与工程成本之间取得了更好的平衡，因此被选为正式方案。

## 10. 项目表达建议

如果后续需要写进项目说明、简历或面试回答，建议这样表达：

### 10.1 简洁版本

> Embedding 模型选型上，项目初期采用 Ollama `qwen3-embedding:4b` 快速完成本地 RAG 闭环，后续引入 `BGE-M3` 做 A/B 检索对比，并从召回率、查询延迟和部署复杂度三个维度评估，最终根据实验结果确定正式方案。

### 10.2 稍完整版本

> 考虑到项目初期重点是快速完成 Python RAG 服务与 Java 企业知识库的对接，我先选择了接入成本更低的 Ollama `qwen3-embedding:4b` 作为 baseline，用于打通文档入库、向量检索和问答链路。后续为了避免主观选型，我计划引入 `BGE-M3` 做对照实验，在相同文档集、相同切块策略和相同 ChromaDB 条件下，对比 Recall@5、MRR@10、入库耗时和查询延迟，再决定正式环境采用哪种方案。
