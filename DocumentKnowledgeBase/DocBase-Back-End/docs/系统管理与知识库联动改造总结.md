# 系统管理与知识库联动改造总结

> 改造日期：2026-05-19  
> 改造范围：Java 后端全量（Controller、Service、Query、SQL、Enum、Permission）  
> 改造目标：让系统管理模块真实支撑知识库业务，不再停留在模板后台层面

---

## 一、改造前后对比

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 菜单权限 | 按钮级权限在 DB 中有定义，但接口只用 page 级 `list` 权限；menu_id 138 被两个脚本冲突占用 | detail/preview/download 分别使用 query/preview/download 权限；冲突解决（add=138, audit=139） |
| 数据权限 | 全模块 binary `isAdmin` 判断，角色 5 级 DataScopeEnum 完全未使用 | KnowledgePermissionHelper 统一封装，ALL/CUSTOM_DEFINE/SINGLE_DEPT/DEPT_TREE/ONLY_SELF 全部生效 |
| 查询过滤 | Controller 私有方法内硬编码 `deptId`/`creatorId`，无部门树感知 | Query 对象支持 `deptIdList`，DEPT_TREE 自动展开子部门 |
| 分类权限 | 仅检查同部门（binary），无数据范围感知 | `ensureCategoryAccessible()` 升级为 5 级数据范围，DEPT_TREE 可操作子部门分类 |
| 审核权限 | 审核人仅限同部门，`auditDocument` 无统一权限入口 | 调用 `getAccessibleDocument()` 完整校验 + `canAccessTargetDept()` 数据范围 |
| 系统配置 | 5 个通用键（皮肤/密码/验证码/注册），无知识库相关项 | +4 个知识库键：上传大小/AI开关/审核开关/默认可见范围 |
| 通知联动 | 审核与通知完全断开 | 审核完成后自动创建通知给文档创建人，异常降级不阻塞 |
| 操作日志 | 部分 Controller 缺少 `@AccessLog`（文档上传无记录） | 6 个知识库写操作全覆盖（上传/下载/审核/分类增删改） |
| 模板残留 | SysNoticeController 有 `/database/slave` 演示端点；SysLogsController 有大段注释死代码；DeptQuery 有 TODO + 注释代码；2 个 Checker 有冗余字段 | 全部清理 |

## 三、架构决策记录

### 决策 1：KnowledgePermissionHelper 放在 admin 模块
- **原因**：Helper 依赖 `DataPermissionCheckerFactory` 和 `DataPermissionService`（均在 admin 模块），无法放入 domain 层
- **影响**：Controller 层使用 Helper 注入 Query 范围；Service 层（domain）通过 `AuthenticationUtils` + `SysDeptService` 直接获取数据范围判断，不跨模块依赖

### 决策 2：Service 层保留 `isAdmin` 参数但不依赖它做权限判断
- **原因**：domain 模块无法依赖 admin 模块的 Helper，但可以通过 `AuthenticationUtils.getSystemLoginUser().getRoleInfo().getDataScope()` 获取数据范围
- **影响**：`getAccessibleDocument()` 等方法的 `isAdmin` 参数保留（向后兼容），但内部权限判断已升级为 `canAccessTargetDept()` 5 级数据范围

### 决策 3：DEPT_TREE 通过 ancestors 字段解析而非递归 SQL
- **原因**：数据库已有 `ancestors` 字段存储祖级路径（如 `0,1,2`），利用此字段做内存过滤比递归 SQL 更简单
- **影响**：`SysDeptServiceImpl.getDeptAndChildrenIds()` 一次性加载所有部门，通过 ancestors 字段匹配子部门

### 决策 4：审核通知异常降级
- **原因**：通知创建是辅助功能，不应阻塞审核主流程
- **影响**：`publishAuditNotice()` 包裹在 try-catch 中，失败仅记录 warn 日志

## 四、可演示场景

### 场景 1：不同数据范围的用户看到不同数据
1. 管理员创建角色"部门主管"，设置 `data_scope=DEPT_TREE`
2. 将用户 A 分配为该角色（所属部门：深圳总公司，dept_id=2）
3. 用户 B（普通用户，SINGLE_DEPT）上传文档到子部门"研发部门"（dept_id=4）
4. 用户 A 登录 → 可看到用户 B 的子部门文档
5. 用户 C（另一个 SINGLE_DEPT 用户，部门=3）→ 看不到该文档

### 场景 2：审核流程 + 通知闭环
1. 普通用户上传文档 → 状态变为"审核中" → 操作日志可查
2. 审核员登录 → 待审核列表中看到该文档 → 点击通过
3. 文档状态变为"已发布"
4. 上传者登录 → 通知公告中出现"您的文档「xxx」已被审批通过"

### 场景 3：系统参数控制业务行为
1. 管理员在参数配置中将 `knowledge.document.auditRequired` 改为 `false`
2. （后续开发接入后）上传文档将跳过审核，直接发布

## 五、明确排除 & 后续建议

本轮改造明确未涉及：
- AI-Assistant Python RAG 后端鉴权
- 前端改动（前端通过菜单树 + 权限接口自动适配）
- `knowledge_document_acl` 细粒度 ACL 表（DDL 已有，Java 层未实现）
- 文档编辑/删除/版本回退功能（Controller 未实现）
- Q&A 问答对接 Python RAG（仍为骨架占位）
- 入库任务自动触发机制

建议后续优先：
1. 实现文档编辑和删除端点（复用已有权限标识 `knowledge:document:edit`/`remove`）
2. 接入 `knowledge.document.auditRequired` 配置到上传流程
3. Python RAG 后端 JWT 鉴权对接
