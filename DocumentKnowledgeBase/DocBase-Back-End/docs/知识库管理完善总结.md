# DocBase 企业文档知识库系统

## 一、这次完善的核心目标

本轮工作的重点，是把“知识库管理”从一个只有菜单和占位页的半成品，推进成可以实际演示的企业文档管理闭环，重点补齐以下几类能力：

1. 文档管理基础闭环
2. 普通用户与管理员的权限分流
3. 文档上传、审批、发布、查看的完整流程
4. 文档详情、版本记录、预览、下载等真实产品能力
5. 为后续 Python 知识库问答助手预留后端接口骨架

## 二、这次完成的主要功能点

### 1. 文档管理闭环

围绕 `知识库管理 -> 文档管理` 完成了最小可用闭环：

- 支持新增文档
- 支持上传本地文件
- 新增后文档状态默认为“审核中”
- 管理员可对文档进行审批通过/驳回
- 审批通过后文档状态变为“已发布”
- 支持列表查询、分页、详情查看

### 2. 权限与可见范围控制

为了让项目更贴近企业内部文档系统，而不是公开资料站，本轮重点做了“谁能看到什么”的隔离逻辑：

- 普通用户默认只能看到自己创建的文档
- 同时可以看到已经发布且对自己可见的共享文档
- 支持三种可见范围：
  - 全员可见
  - 本部门可见
  - 仅本人可见
- 管理员保留审批能力

这部分改造的意义在于，项目不再只是“文件上传列表”，而是具备了企业知识库场景中最关键的数据隔离能力。

### 3. 文档详情与版本信息

文档详情页补充了完整的业务信息展示：

- 文档标题、编号、分类、创建人、所属部门
- 当前状态、可见范围、审核备注
- 当前版本信息
- 历史版本记录

这样可以直接体现“文档型系统”的产品特征，而不是普通 CRUD 表格。

### 4. 文档预览与下载

这部分是真正把“文档”做成“可使用产品”的关键能力：

- 支持文档预览
- 支持文档下载
- 文件来源于上传后的当前版本文件
- 下载最终改为前端拉取 `blob` 后强制保存，避免浏览器只打开 PDF 而不下载

### 5. 审核记录表与审核历史展示

为了让审批流程可追踪，本轮增加了审核记录能力：

- 设计并补充 `knowledge_document_audit_log` 审核记录表升级 SQL
- 审批时写入审核记录
- 文档详情页展示审核历史
- 包括审核人、审核结果、审核前状态、审核后状态、审核备注、审核时间

这部分非常适合作为项目亮点讲给面试官，因为它体现了“流程留痕”和“业务可追溯”设计。

### 6. 知识库问答助手后端骨架

为后续接入 Python RAG / LLM 服务，预留了知识库问答接口骨架：

- `GET /knowledge/qa/health`
- `POST /knowledge/qa/ask`

当前先返回占位结果，但接口边界已经提前定义好，后续可以直接对接 Python 服务，不需要再大改前后端结构。

## 三、这次改造过程中遇到的主要困难

### 1. 开源模板遗留逻辑较多，知识库模块不是原生主线

原项目本质上是一个后台管理模板，知识库相关能力并不是原本的成熟主业务，所以前期遇到了几个问题：

- 菜单能显示，但页面大多只是占位
- 权限按钮、菜单权限、动态路由之间并没有天然闭环
- 前后端有不少模板残留逻辑，不完全适配知识库业务

#### 解决方案

- 先从菜单和权限入手，保证知识库页面可以真正显示
- 再补“最小骨架”，而不是一开始追求大而全
- 按“能上传、能审批、能查看、能下载”的主链路逐步落地

### 2. 普通用户看不到共享文档

这是本轮比较典型的业务逻辑问题。最开始普通用户只能看到自己创建的文档，即使管理员已经审批通过并设置为“全员可见”，普通用户依然看不到。

#### 原因

控制器里对普通用户查询做了强制 `creatorId = currentUserId` 限制，导致共享文档也被拦掉了。

#### 解决方案

重构文档查询逻辑，引入：

- `onlyVisibleToCurrentUser`
- `includePublishedShared`
- `deptId`

最终实现为：

- 普通用户可查看自己的文档
- 也可查看已发布且符合可见范围的共享文档
- 本部门可见文档增加部门过滤

### 3. 文档详情因为新表未建而报数据库异常

新增“审核历史展示”后，文档详情页会查询 `knowledge_document_audit_log`。如果数据库尚未执行升级 SQL，详情接口会直接抛数据库异常，导致“详情 / 预览 / 下载”一起失败。

#### 解决方案

做了兼容降级：

- 审核历史查询异常时自动返回空列表
- 不再影响详情主流程

这样可以保证系统在数据库未完全升级时，主功能仍可继续使用。

### 4. 下载按钮只会在浏览器中打开 PDF，不会真正下载

这是本轮比较典型的前端交互问题。

#### 原因

最初直接使用文件 URL，新标签页会按浏览器默认行为打开 PDF，而不是强制下载。

#### 解决方案

前端改成：

1. 先通过文件 URL 拉取二进制 `blob`
2. 再通过 `a` 标签 + `download` 属性触发本地保存

这样就从“浏览器预览行为”切换成了“明确下载行为”。

### 5. 前端遗留类型错误较多

在补知识库主链路时，还暴露出系统管理部分的历史 TypeScript 模板错误，尤其是日志页、角色页这类旧页面。

#### 解决方案

- 逐个重写有问题的页面模板
- 修正 `date-picker`、`sort-change`、分页事件等类型绑定
- 最终让前端 `pnpm typecheck` 重新通过

这样后续继续优化系统管理时，基础状态会更稳定。

## 四、这次改造的技术要点

### 1. 业务状态设计

文档状态按业务流程划分为：

- 草稿
- 审核中
- 已发布
- 已驳回
- 已归档

这比“单纯上传文件”更符合企业知识库的管理场景。

### 2. 数据权限设计

可见范围并不是简单字段展示，而是真正参与了查询与详情权限判断：

- 列表层控制哪些文档能查到
- 详情层再次校验是否有权限查看

这种“双层控制”更安全，也更接近真实项目实践。

### 3. 文件与业务数据分离

文档主表保存业务属性：

- 标题
- 编号
- 状态
- 可见范围
- 分类

文档版本表保存文件属性：

- 文件名
- 文件大小
- 存储路径
- 版本号
- 当前版本标记

这样后续如果做“版本迭代上传”“解析任务”“全文检索”，扩展性更好。

### 4. 流程留痕

审批不只是更新状态，还额外记录审批历史，形成：

- 当前状态
- 审核备注
- 审核历史表

这让系统具备了基础审计能力。

## 五、这次改造后，知识库模块已经具备的产品能力

从演示角度看，当前知识库管理已经不再是简单后台模板页面，而是具备以下较完整的业务表达：

- 分类管理
- 文档新增与文件上传
- 文档列表
- 文档详情
- 版本信息查看
- 审核流转
- 审核历史
- 普通用户数据隔离
- 文档预览与下载
- 知识库问答后端接口预留
