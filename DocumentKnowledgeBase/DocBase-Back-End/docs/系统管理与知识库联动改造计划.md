# DocBase 系统管理与知识库联动改造计划

## Context

当前 DocBase 已经具备知识库分类、文档、审核、下载、AI 入口等基础能力，但“系统管理”与“知识库管理”的联动还不够完整。现状上仍有以下典型问题：

1. 知识库模块的菜单、按钮、接口权限点还没有完全和系统管理中的 RBAC 配置对齐。
2. 知识库模块的数据权限仍有较多业务内局部判断，尚未稳定接入现有 `DataScopeEnum + Checker + @dataScope` 体系。
3. 用户、角色、部门、参数配置、日志审计等系统能力还没有充分支撑知识库业务。
4. 历史 SQL、菜单初始化、权限标识可能存在残留冲突，需要统一清理。

本次改造目标不是泛化重构，而是让系统管理模块真实支撑知识库模块，并且不破坏当前已经可用的知识库能力。

---

## 阶段 1：菜单、权限点、SQL 初始化与接口鉴权校准

**阶段目标**  
统一知识库相关菜单、按钮、接口权限点定义，避免 `sys_menu`、前端按钮鉴权、后端 `@PreAuthorize` 三者不一致。

### 1.1 排查 SysNoticeController 的历史兼容问题

- **文件位置**：`docbase-admin/.../controller/system/SysNoticeController.java`
- **任务内容**：
  - 检查是否存在不再适配当前项目的数据源或历史逻辑。
  - 如存在与知识库联动相关的阻塞问题，修复；否则不作为本阶段主线改造重点。
- **注意**：
  - 不要为了本次知识库联动去无边界重写通知模块。

### 1.2 排查 SysLogsController 的历史兼容问题

- **文件位置**：`docbase-admin/.../controller/system/SysLogsController.java`
- **任务内容**：
  - 检查日志管理相关控制器是否存在历史兼容问题、接口异常或权限缺失。
  - 本阶段仅做阻塞项清理，不做日志体系大改。

### 1.3 排查历史菜单 ID / 权限初始化冲突

- **重点说明**：
  - 历史脚本里可能存在菜单 ID、按钮 ID、角色授权冲突。
  - 本任务是“排查并清理历史冲突”，**不是在新代码中继续依赖固定 menu_id / role_id**。

- **文件位置**：
  - `sql/knowledge-document-audit-upgrade.sql`
  - `sql/knowledge-base-reconciliation.sql`
  - 其他知识库菜单初始化脚本

- **任务内容**：
  - 梳理知识库菜单、按钮、角色授权初始化脚本之间的关系。
  - 找出重复、冲突、历史遗留未统一的菜单或按钮数据。
  - 输出统一后的菜单/按钮初始化脚本。

- **强约束**：
  - 不允许新逻辑写死 `menu_id=138`、`role_id=2` 这类实现依赖。
  - 可以记录历史冲突来源，但最终方案必须按“权限标识”而不是固定 ID 工作。

### 1.4 校准 sys_menu DDL / 初始化数据描述

- **文件位置**：
  - `sql/docbase_knowledge.sql`
  - 其他菜单初始化 SQL

- **任务内容**：
  - 清理菜单类型、路由类型、按钮类型描述与实际使用不一致的问题。
  - 保证初始化 SQL、数据库注释、前端菜单渲染逻辑表达一致。

### 1.5 系统梳理知识库 Controller 的 `@PreAuthorize`

- **重点说明**：
  - 本任务是知识库权限校准的主线。
  - 需要逐个核对接口权限，避免“前端有入口、后端 403”或“后端放开但前端没做按钮控制”。

- **重点排查对象**：
  - `KnowledgeDocumentController`
    - `list()`
    - `detail()`
    - `preview()`
    - `download()`
    - `add()`
  - `KnowledgeCategoryController`
    - `list()`
    - `detail()`
    - `add()`
    - `edit()`
    - `remove()`
  - `KnowledgeDocumentAuditController`
  - `KnowledgeIngestTaskController`
  - `AiChatController`
  - `AiAuditController`
  - `KnowledgeQaController`

- **特别说明**：
  - `health()` 这类接口要先区分是业务接口还是运维/调试接口，再决定是否纳入业务权限。
  - `detail / preview / download` 不应简单复用同一个粗粒度权限，应明确是否拆分为 `query / preview / download`。

### 1.6 补齐知识库关键操作的 `@AccessLog`

- **目标**：
  - 让系统管理中的操作日志能真实记录知识库关键行为。

- **重点补齐对象**：
  - `KnowledgeDocumentController.add()`
  - `KnowledgeCategoryController.add/edit/remove`
  - `KnowledgeDocumentAuditController.audit()`
  - 其他明确属于关键业务操作的接口

- **注意**：
  - 只记录关键业务动作，不要为了“全记录”导致日志噪音过大。

### 阶段 1 验收标准

1. 管理员和普通用户登录后，知识库菜单和按钮显示符合权限配置。
2. 普通用户不会因历史菜单脚本问题拿到越权按钮。
3. 前端权限控制和后端 `@PreAuthorize` 一致。
4. 关键知识库写操作能写入 `sys_operation_log`。

## 阶段 2：从散落的 isAdmin 判断，逐步收敛到统一的数据权限辅助层

**阶段目标**  
减少知识库模块中散落的 `isAdmin + deptId + creatorId` 业务判断，将可复用的数据权限逻辑抽取为统一辅助能力，但不做脱离现状的大爆炸式重构。

### 2.1 新增 `KnowledgePermissionHelper`

- **文件建议**：`docbase-admin/.../service/permission/KnowledgePermissionHelper.java`

- **职责建议**：
  - `applyDocumentQueryScope(KnowledgeDocumentQuery query)`
  - `applyCategoryQueryScope(KnowledgeCategoryQuery query)`
  - `applyIngestTaskQueryScope(KnowledgeIngestTaskQuery query)`
  - `checkDocumentAccessible(KnowledgeDocumentEntity entity)`
  - `checkCategoryAccessible(KnowledgeCategoryEntity entity)`
  - `checkDocumentAuditable(KnowledgeDocumentEntity entity)`
  - `resolveTargetDeptId(Long commandDeptId)`

- **重要约束**：
  - 第一批优先覆盖：
    - `KnowledgeDocument`
    - `KnowledgeCategory`
    - `KnowledgeIngestTask`
  - `AiChatController`、`AiAuditController` 可列为第二批接入，不强制与本阶段主线绑死。

### 2.2 Query 层增加数据范围字段

- **建议排查对象**：
  - `KnowledgeDocumentQuery`
  - `KnowledgeCategoryQuery`
  - `KnowledgeIngestTaskQuery`

- **建议内容**：
  - 增加适配数据权限的查询条件，例如：
    - `List<Long> deptIdList`
    - `Long creatorId`
    - 必要的辅助布尔标记

- **重要提醒**：
  - 文档查询**不能仅依赖 `dept_id in (...)`**。
  - 必须同时考虑：
    - `creator_id`
    - `visibility`
    - `status`
    - “创建人自己永远可见”
    - “仅本人可见 / 本部门可见 / 全员可见”

### 2.3 在 SysDeptService 中补齐部门树能力

- **建议方法**：
  - `List<Long> getDeptAndChildrenIds(Long deptId)`

- **用途**：
  - 支撑 `DEPT_TREE` 数据范围。

- **注意**：
  - 只提供稳定的部门树能力，不要把知识库业务规则直接硬编码进部门服务。

### 2.4 用 Helper 替换 Controller 中散落的 `restrictToCurrentUserIfNeeded`

- **重点对象**：
  - `KnowledgeDocumentController`
  - `KnowledgeCategoryController`
  - `KnowledgeIngestTaskController`

- **建议做法**：
  - 将列表查询前的权限范围注入收敛到 `KnowledgePermissionHelper.apply*QueryScope()`。

- **注意**：
  - 不要求本阶段把所有控制器一次性全替换完。
  - 优先处理知识库主链路，避免大面积重构导致回归范围失控。

### 2.5 收敛 ApplicationService 中散落的 `isAdmin` 业务判断

- **重点对象**：
  - `KnowledgeDocumentApplicationService`
  - `KnowledgeCategoryApplicationService`

- **建议方向**：
  - 把重复的权限判断沉淀到 `KnowledgePermissionHelper` 或相似辅助层。

- **重要说明**：
  - 目标是“减少散落的 `isAdmin` 判断”，不是强制做到“零 `isAdmin`”。
  - 必要的管理员快速通道可以保留，只要逻辑集中、清晰、可测试即可。

### 阶段 2 验收标准

1. 文档、分类、入库任务的查询范围规则更统一。
2. 文档权限判断没有被简化成单一部门过滤，现有可见性规则不被破坏。
3. 管理员、部门范围、仅本人、公开文档等场景都能正确工作。
4. 代码中散落的权限判断明显减少，但没有因为“强行统一”引入行为回归。

## 阶段 3：系统配置、通知、日志 与知识库业务联动

**阶段目标**  
让系统管理中的配置、通知、日志真正服务知识库业务，但避免本轮做过深的基础设施改造。

### 3.1 增加知识库相关系统参数

- **建议位置**：
  - `ConfigKeyEnum`
  - `sys_config` 初始化 SQL

- **建议参数**：
  - `knowledge.document.maxUploadSizeMb`
  - `knowledge.ai.chatEnabled`
  - `knowledge.document.auditRequired`
  - `knowledge.document.defaultVisibility`

- **注意**：
  - 本轮优先完成“配置项定义 + 读取骨架 + 关键业务接入点”。
  - 如果上传工具层动态化改造成本过高，可以先不强制替换掉当前静态常量实现，避免本轮范围过大。

### 3.2 审核结果与通知能力联动

- **重点对象**：
  - `KnowledgeDocumentApplicationService.auditDocument()`

- **任务内容**：
  - 审核通过/驳回后，为后续通知能力预留统一调用点。
  - 若当前项目已有通知/公告应用服务，可复用；若没有，先保留清晰扩展点，不强行发散实现。

- **注意**：
  - 不要把通知逻辑硬编码得过深，避免耦合。

### 3.3 让知识库关键动作可进入系统日志

- **重点动作**：
  - 上传文档
  - 审核文档
  - 分类新增/编辑/删除
  - 必要时记录下载行为

- **注意**：
  - 下载日志是否纳入操作日志，需要结合系统现有日志粒度决定，不强制过度记录。

### 阶段 3 验收标准

1. 系统参数中能看到知识库关键配置项。
2. 审核流程具备通知/公告联动扩展点。
3. 知识库关键动作能够进入系统日志体系。

## 禁止事项

1. 不允许新代码依赖固定 `menu_id`、`role_id`、`dept_id`、用户名。
2. 不允许只改前端菜单/按钮，不改后端鉴权。
3. 不允许为了统一权限，破坏现有知识库的 `visibility + creator + status` 组合规则。
4. 不允许把本轮改造成无边界的大重构。
5. 不允许删除当前已经可用的知识库功能。

---

## 回归重点

1. 管理员登录后的知识库菜单、按钮、接口访问。
2. 普通用户登录后的知识库菜单、按钮、接口访问。
3. 全员可见文档的列表、详情、预览、下载。
4. 本部门可见文档在同部门与跨部门账号下的差异。
5. 仅本人可见文档的访问控制。
6. 分类管理的增删改查权限。
7. 文档审核列表、审核通过、审核驳回。
8. AI 问答、AI 审计入口的菜单和接口权限。
9. 关键知识库操作是否进入系统日志。

---

## 风险提示

1. 如果本轮强推所有知识库子模块同时接入统一 Helper，容易把回归范围拉得过大。
2. 如果把数据权限简单理解成部门过滤，会破坏知识库现有可见性模型。
3. 如果继续依赖历史固定菜单 ID，会导致后续环境初始化和权限回归持续不稳定。
4. Python RAG 侧如果不在当前仓库内，本轮只考虑接口兼容，不强行纳入开发范围。
