#### 2.1 文档可见范围（visibility）枚举值定义前后端三方矛盾

**文件：**
- SQL 建表：`sql/knowledge-base-init.sql:34` — `visibility smallint DEFAULT '2' COMMENT '可见范围（1私有 2部门 3公开 4自定义）'`
- 后端逻辑：[KnowledgeDocumentApplicationService.java:202-211](docbase-domain/src/main/java/com/docbase/domain/knowledge/document/KnowledgeDocumentApplicationService.java#L202-L211) — `equals(entity.getVisibility(), 1)` 判定为"全员可见"
- 前端展示：[document/index.vue:108-112](DocBase-Front-End/src/views/knowledge/document/index.vue#L108-L112) — `1: "全员可见", 2: "本部门可见", 3: "仅本人可见"`

**问题：** SQL 注释说明 1=私有、3=公开；后端代码将 1 视为公开（全员可见）；前端也将 1 视为全员可见。**三方定义完全矛盾**，数据库按照原注释写入数据后，代码层会按反义解读，导致权限完全逆向——私有文档变为公开，公开文档变为私有。

**触发场景：**
1. 管理员按数据库注释理解，将 visibility 设为 1 期望仅自己可见
2. 后端 `canCurrentUserView()` 方法判定 1=全员可见，该文档对所有人开放
3. 反之，visibility 设为 3 期望是公开文档，但代码中没有处理 visibility==3 的逻辑（默认不可见）

**影响范围：** 所有文档的可见范围权限完全失效或反向生效。

#### 2.2 文档状态（status）枚举值前后端三方矛盾

**文件：**
- SQL 建表：`sql/knowledge-base-init.sql:34` — `status smallint DEFAULT '1' COMMENT '文档状态（1草稿 2已上传 3入库中 4生效 5失败 6归档）'`
- 后端逻辑：[KnowledgeDocumentApplicationService.java:202-207](docbase-domain/src/main/java/com/docbase/domain/knowledge/document/KnowledgeDocumentApplicationService.java#L202-L207) — `equals(entity.getStatus(), 3)` 判定为"已发布"；`equals(entity.getStatus(), 2)` 判定为"审核中"
- 前端展示：[document/index.vue:97-106](DocBase-Front-End/src/views/knowledge/document/index.vue#L97-L106) — 1=草稿, 2=审核中, 3=已发布, 4=已驳回, 5=已归档

**问题：** SQL schema 定义的状态机（草稿→已上传→入库中→生效→失败→归档）与代码实现的审批工作流（草稿→审核中→已发布→已驳回→已归档）完全不同。按SQL定义，status=3 表示"入库中"，但代码将其作为"已发布"展示给用户。

**触发场景：**
1. 文档入库任务将状态更新为 3（入库中，按SQL定义）
2. 前端将 status=3 渲染为"已发布"绿色标签
3. `canCurrentUserView()` 将 status=3 视为"可公开访问"的条件之一
4. 入库中的文档被错误地视为已发布并公开可见

**影响范围：** 文档生命周期状态机完全不可信，审核流与入库流互相污染。

#### 2.5 文档审核无部门级别的越权控制

**文件：** [KnowledgeDocumentApplicationService.java:215-239](docbase-domain/src/main/java/com/docbase/domain/knowledge/document/KnowledgeDocumentApplicationService.java#L215-L239)

2. 传入 B 部门的私密文档 ID
3. 直接通过/驳回，操作 B 部门文档完全不受限

#### 2.7 前端下载直接使用原始 URL 绕过权限检查

**文件：** [document/index.vue:333-374](DocBase-Front-End/src/views/knowledge/document/index.vue#L333-L374)

**关联风险：** `SecurityConfig.java:99-100` 配置 `"/profile/**").permitAll()`——静态文件目录允许匿名访问，这意味着文件 URL 即使不带 Token 也能被直接下载。

#### 2.8 文件上传仅做扩展名校验，存在伪装绕过风险

**文件：** [FileUploadUtils.java:135-152](docbase-common/src/main/java/com/docbase/common/utils/file/FileUploadUtils.java#L135-L152)

```java
static void isAllowedUpload(MultipartFile file, String[] allowedExtension) {
    // 仅检查文件名长度和扩展名
    String extension = getFileExtension(file);
    if (!isExtensionAllowed(extension, allowedExtension)) {
        throw new ApiException(...);
    }
}
```

**问题：**
1. **仅检查扩展名，不检查文件内容（Magic Number）**：攻击者可将 `.jsp` 文件改名为 `.pdf` 上传
2. **允许 ZIP/RAR 格式**：可上传含恶意内容的压缩包，解压后可能触发 Zip Slip 漏洞
3. **扩展名获取回退到 Content-Type**：`getFileExtension()` 在无扩展名时读取 MIME 类型——这是攻击者可完全控制的 HTTP 头
4. **存储路径保留原始文件名**：`generateFilename` 包含原始文件名的 BaseName，可能引入路径遍历字符

**触发场景（Zip Slip）：**
1. 攻击者构造含 `../../etc/cron.d/malicious` 路径条目的 ZIP 文件
2. 上传为 `payload.zip`，通过扩展名字段
3. 如果后续入库解析模块解压该文件，可能覆盖系统关键文件

#### 2.11 文档审核存在并发竞态条件

**文件：** [KnowledgeDocumentApplicationService.java:215-239](docbase-domain/src/main/java/com/docbase/domain/knowledge/document/KnowledgeDocumentApplicationService.java#L215-L239)

#### 2.12 文档创建时 currentVersionId 可能为空

**文件：** [KnowledgeDocumentApplicationService.java:243-277](docbase-domain/src/main/java/com/docbase/domain/knowledge/document/KnowledgeDocumentApplicationService.java#L243-L277)

```java
knowledgeDocumentService.save(entity);           // Step 1: 保存文档
// ...
knowledgeDocumentVersionService.save(versionEntity); // Step 2: 保存版本

try {
    auditHistoryList = knowledgeDocumentAuditLogService.list(...);
} catch (Exception exception) {
    log.warn("加载文档审核历史失败，已自动降级为空列表，documentId={}", documentId, exception);
    auditHistoryList = Collections.emptyList();
}
```

**问题：** `catch (Exception exception)` 捕获了所有异常（包括 NPE、DB 连接池耗尽等严重错误），用户看到的只是审核历史为空，无法区分"确实没有审核历史"和"数据库挂了"。
