# DocBase 后端项目结构说明

## 1. 文档目的

这份文档用于说明 `DocBase-Back-End` 当前的后端工程结构，帮助快速理解：

- 整个项目为什么不是传统 `controller / service / dao / mapper` 全部平铺在一个模块里
- 每个 Maven 模块分别负责什么
- 模块内部每个目录、子目录通常存放哪些代码
- 一个接口请求大致会经过哪些目录和类

当前项目更适合归类为：

- **Maven 多模块**
- **模块化单体架构**
- **按职责拆分模块，模块内再分层**

它不是微服务，也不是传统课程里常见的“单工程三层结构”。

---

## 2. 根目录结构

`DocBase-Back-End` 根目录下主要包含这些内容：

### 2.1 `.github`

用于放 GitHub 相关配置，例如：

- Issue 模板
- CI / CD 工作流

这部分不参与运行时业务逻辑。

### 2.2 `.idea`

IDEA 的项目配置目录，包含：

- 编码设置
- 项目模块索引
- 数据源配置

这部分是开发环境配置，不属于运行时代码。

### 2.3 `.mvn`

Maven Wrapper 相关文件目录，用于支持：

- 直接执行 `./mvnw`
- 保证团队成员本地 Maven 版本使用方式一致

### 2.4 `bin`

项目启动脚本目录，通常包括：

- Windows 启动脚本
- Shell 启动脚本

适合部署或本地快速启动时使用。

### 2.5 `docker`

用于放运行依赖说明或容器相关资料，例如：

- MySQL 说明
- Redis 说明

这部分偏部署辅助，不是业务代码。

### 2.6 `docs`

项目文档目录，用来放：

- 改造说明
- 结构说明
- 设计文档
- 方案记录

你现在正在看的这份文档也放在这里。

### 2.7 `sql`

数据库脚本目录，当前主要包含：

- 主初始化 SQL
- 版本 SQL
- 组合脚本

后续如果要重建数据库、准备演示环境、补初始化脚本，优先看这里。

### 2.8 `pom.xml`

根 Maven 配置文件，负责：

- 声明整个工程是多模块项目
- 统一依赖版本
- 统一插件版本
- 声明公共构建参数

这相当于整个后端工程的总控配置。

## 3. Maven 模块总览

后端当前主要由 4 个核心模块组成：

- `docbase-admin`
- `docbase-common`
- `docbase-domain`
- `docbase-infrastructure`

可以先这样粗略理解：

- `docbase-admin`：接口入口层 / Controller 层
- `docbase-common`：公共基础模块
- `docbase-domain`：核心业务层
- `docbase-infrastructure`：基础设施和框架支撑层

---

## 4. `docbase-admin` 模块

### 4.1 模块职责

`docbase-admin` 是后端对外提供 HTTP 接口的入口模块，最接近你在 JavaWeb 课程里熟悉的 `controller` 层。

这里主要负责：

- 接收前端请求
- 参数绑定
- 调用领域层服务
- 返回统一响应对象
- 处理登录、权限、访问日志等 Web 相关逻辑

### 4.2 目录说明

#### `src/main/java/com/docbase/admin/controller`

这是控制器目录，真正的接口类主要都在这里。

按业务又分为：

- `controller/common`
  - 放公共接口
  - 例如登录、验证码、获取当前用户信息、注册、路由获取等

- `controller/system`
  - 放系统管理相关接口
  - 例如用户、角色、菜单、部门、参数、日志、公告等

- `controller/knowledge`
  - 放知识库相关接口
  - 例如分类、文档、导入任务等

- `controller/ai`
  - 放 AI 相关接口
  - 例如 AI 问答、AI 审计

这部分就是你最熟悉的 `Controller`。

#### `src/main/java/com/docbase/admin/customize`

这个目录可以理解为 `admin` 模块下的“Web 侧定制能力”。

它不是单纯接口，而是围绕后台系统运行的辅助逻辑。

主要包括：

- `aop`
  - AOP 切面能力
  - 例如访问日志记录

- `async`
  - 异步任务封装
  - 比如操作日志、登录日志异步落库相关逻辑

- `config`
  - 安全配置、过滤器注册、认证链条等

- `service`
  - 这里不是领域业务服务
  - 而是偏“后台登录态、安全、权限”的支撑服务
  - 常见如：
    - 登录服务
    - Token 服务
    - 用户认证服务
    - 数据权限检查器

### 4.3 `src/main/resources`

主要包含 admin 模块自己的配置文件：

- `application.yml`
  - 总体启动配置
- `application-dev.yml`
  - 开发环境配置
- `application-test.yml`
  - 测试环境配置

其中数据库连接、Redis、日志目录等通常会在这里引用或覆盖。

### 4.4 你可以怎么理解这个模块

如果映射到课程项目：

- `docbase-admin`
  = `controller` 层
  + 一部分安全配置
  + 一部分登录与权限相关辅助服务

## 5. `docbase-common` 模块

### 5.1 模块职责

`docbase-common` 是全项目共用的基础代码模块。

它主要负责放：

- 公共对象
- 公共常量
- 公共异常
- 公共枚举
- 公共工具类

这个模块不直接承载具体业务，但会被所有其他模块依赖。

### 5.2 目录说明

#### `annotation`

放自定义注解，例如：

- Excel 导出相关注解
- 通用辅助注解

#### `config`

放公共配置类，例如：

- 全局项目配置对象

#### `constant`

放系统常量定义，例如：

- 通用常量
- 状态常量
- 缓存 Key 常量

#### `core`

这是公共核心对象目录，通常很重要。

里面又会细分：

- `core/base`
  - 放基础父类
  - 例如 `BaseController`、基础实体父类等

- `core/dto`
  - 放统一返回对象
  - 例如 `ResponseDTO`

- `core/page`
  - 放分页相关抽象和返回对象
  - 例如：
    - 抽象查询对象
    - 抽象分页对象
    - `PageDTO`

#### `enums`

放系统枚举。

通常又分：

- `enums/common`
  - 通用业务枚举
  - 例如状态、菜单类型、登录状态、操作类型等

- `enums/dictionary`
  - 字典相关枚举和映射

#### `exception`

放异常体系。

通常包括：

- 自定义业务异常
- 错误码定义
- 错误码接口

#### `utils`

放公共工具类。

按功能又分为：

- `utils/file`
  - 文件处理
- `utils/i18n`
  - 国际化
- `utils/ip`
  - IP 地址与归属地解析
- `utils/jackson`
  - JSON 工具
- `utils/poi`
  - Excel / POI 工具
- `utils/time`
  - 时间工具

### 5.3 你可以怎么理解这个模块

如果映射到课程项目：

- `docbase-common`
  = `common + utils + base + enums + exception`

## 6. `docbase-domain` 模块

### 6.1 模块职责

`docbase-domain` 是当前项目最核心的业务模块。

它和你在课程里接触的 `service + mapper + entity + dto + query` 最接近，但组织方式更工程化。

这个模块负责：

- 组织业务应用服务
- 定义业务命令与查询对象
- 定义 DTO
- 封装业务模型
- 存放数据库实体、Mapper、数据库服务

### 6.2 模块组织方式

`domain` 不是按“统一的 service / mapper / entity”平铺，而是先按业务域拆分。

当前主要业务域有：

- `common`
- `system`
- `knowledge`
- `ai`

#### 6.2.1 `common`

放领域层共用能力，例如：

- `cache`
  - 领域缓存中心
  - Redis / Guava / Map 缓存抽象

- `command`
  - 通用命令对象
  - 例如批量操作命令

- `dto`
  - 公共 DTO
  - 如当前登录用户信息、Token、上传结果等

#### 6.2.2 `system`

放传统后台系统能力，对应：

- `config`
- `dept`
- `log`
- `menu`
- `monitor`
- `notice`
- `post`
- `role`
- `user`

每个业务子目录内部通常都类似：

- `xxxApplicationService`
  - 业务应用服务
  - 类似你熟悉的 `Service`

- `command`
  - 新增、修改、状态变更等写操作命令对象

- `query`
  - 查询条件对象

- `dto`
  - 返回给 controller / 前端 的数据传输对象

- `model`
  - 业务模型
  - 用来放领域规则、校验、封装复杂业务逻辑

- `db`
  - 数据库访问相关代码
  - 这里最接近传统的 `entity + mapper + dao`

#### 6.2.3 `knowledge`

这是知识库业务域，目前是本项目重点改造方向。

当前已拆出的知识库子域包括：

- `category`
  - 分类管理

- `document`
  - 文档主表、版本、文档元数据等

- `ingest`
  - 入库任务

- `permission`
  - 文档权限控制

这些目录未来会承接真正的企业文档知识库核心业务。

#### 6.2.4 `ai`

AI 相关业务域，当前主要有：

- `ai/chat`
  - AI 问答会话与消息

- `ai/audit`
  - AI 审计日志

这些目录为后续接入 RAG / Python 服务调用打基础。

### 6.3 `db` 子目录到底是什么

这是你最容易和课程项目对照的地方。

在每个业务子域下的 `db`，通常会放：

- `SysXxxEntity`
  - 数据库实体对象

- `SysXxxMapper`
  - MyBatis / MyBatis Plus Mapper

- `SysXxxService`
  - 数据访问服务接口

- `SysXxxServiceImpl`
  - 数据访问服务实现

所以：

- 这里的 `Mapper`
  基本就相当于你熟悉的 `DAO`

### 6.4 你可以怎么理解这个模块

如果映射到课程项目：

- `docbase-domain`
  = `service + dto + query + entity + mapper + 业务模型`

它只是没有平铺，而是按业务域打包。

## 7. `docbase-infrastructure` 模块

### 7.1 模块职责

`docbase-infrastructure` 是基础设施层。

它不直接负责业务，而是负责：

- 框架适配
- 配置封装
- 缓存实现
- 安全工具
- 异常处理
- 线程池
- 存储接入
- MyBatis Plus 支撑

如果说 `domain` 是“业务脑子”，那 `infrastructure` 更像“技术底盘”。

### 7.2 目录说明

#### `annotations`

放基础设施层的通用注解，例如：

- 限流注解
- 数据权限相关注解

#### `cache`

放缓存层相关实现与工具。

#### `client`

放外部客户端调用封装，适合后续接入第三方服务或 Python 服务。

#### `config`

放各种框架配置类，是非常核心的一块。

通常包括：

- 数据源配置
- Redis 配置
- Swagger / OpenAPI 配置
- Captcha 配置
- 多数据源配置

#### `exception`

放全局异常处理，例如：

- 全局异常拦截
- 数据库异常切面
- Web 异常处理

#### `filter`

放过滤器，例如：

- TraceId 过滤器
- Web 请求链路过滤器

#### `i18n`

放国际化辅助能力，例如：

- i18n 检查器

#### `log`

放日志切面和日志增强能力。

#### `minio`

放对象存储接入代码，后续文档上传、版本文件管理通常会用到这里。

#### `mybatisplus`

放 MyBatis Plus 相关能力，例如：

- 自动填充
- 代码生成器
- MySQL 函数扩展

#### `schedule`

放调度任务管理相关代码。

#### `security`

放安全工具，如：

- RSA 加密
- XSS 防护

#### `thread`

放线程池和线程管理相关配置。

#### `user`

放登录上下文和当前用户工具，例如：

- `AuthenticationUtils`
- 登录用户对象
- 数据权限相关对象

### 7.3 `src/main/resources`

这里主要放底层资源和基础配置。

常见内容包括：

- `application-basic.yml`
  - 基础配置

- `banner.txt`
  - Spring Boot 启动 Banner

- `logback-spring.xml`
  - 日志配置

- `i18n/messages.properties`
  - 国际化资源

- `h2sql`
  - H2 初始化 SQL

- `pgsql`
  - PostgreSQL 初始化 SQL

### 7.4 你可以怎么理解这个模块

如果映射到课程项目：

- `docbase-infrastructure`
  = `config + framework + security + cache + persistence support`

这部分通常在课程项目里会比较弱，但在正式项目里会比较重。

## 8. 一次请求的大致调用链

以“用户管理列表查询”为例，通常调用链是：

1. 前端请求进入 `docbase-admin`
2. `SysUserController` 接口接收参数
3. 调用 `docbase-domain` 中的 `UserApplicationService`
4. `UserApplicationService` 组织业务逻辑
5. 再调用 `system/user/db` 下的数据访问对象
6. `Mapper` 查询数据库
7. 查询结果转换成 `DTO`
8. `controller` 返回 `ResponseDTO`

所以这个项目并不是没有：

- Controller
- Service
- Mapper
- Entity

而是它们被拆散到了不同模块和业务目录里。

## 9. 和传统 JavaWeb 三层结构的对照

如果对照你在课程里学过的结构，可以这样看：

### 传统课程结构

```text
controller
service
serviceImpl
mapper / dao
entity
vo / dto
```

### 当前项目的大致对应

- `docbase-admin/controller`
  = `controller`

- `docbase-domain/*ApplicationService`
  = `service`

- `docbase-domain/**/db/*Mapper`
  = `mapper / dao`

- `docbase-domain/**/db/*Entity`
  = `entity`

- `docbase-domain/**/dto`
  = `dto`

- `docbase-domain/**/query`
  = `query / vo`

- `docbase-domain/**/command`
  = 写操作请求对象

- `docbase-common`
  = 公共基础代码

- `docbase-infrastructure`
  = 配置与框架支撑层

---

## 10. 当前结构的优点

这套结构相比课程里的简单三层有几个明显优点：

- 模块职责更清晰
- 公共代码和业务代码分离更彻底
- 系统模块、知识库模块、AI 模块更容易继续扩展
- 便于后续做权限、存储、缓存、日志等横切能力
- 对中大型项目更友好

---

## 11. 当前结构的理解重点

如果你刚接触这套结构，建议先牢牢记住这三句话：

1. `docbase-admin` 主要就是接口入口层  
2. `docbase-domain` 才是真正的业务核心层  
3. `docbase-infrastructure` 是技术底盘，不是业务层

再进一步理解：

- 传统 `dao` 在这个项目里更多表现为 `Mapper`
- `service` 在这个项目里更多表现为 `ApplicationService`
- `model` 是比普通课程项目更进阶的一层，用来沉淀业务规则

---

## 12. 后续阅读建议

如果想真正把这套结构看顺，建议按下面顺序读代码：

1. 先看 `docbase-admin/controller/common/LoginController`
2. 再看 `docbase-admin/controller/system/SysUserController`
3. 顺着进入 `docbase-domain/system/user/UserApplicationService`
4. 再看 `docbase-domain/system/user/db`
5. 最后看 `docbase-infrastructure/user` 和 `docbase-infrastructure/config`

这样最容易把“请求入口 -> 业务处理 -> 持久化 -> 登录上下文”这条主线串起来。
