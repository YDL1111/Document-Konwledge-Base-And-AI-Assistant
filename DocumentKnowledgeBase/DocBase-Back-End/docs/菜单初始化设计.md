# 菜单初始化设计

## 1. 文档目的

本文档用于设计企业文档知识库系统接入现有 `sys_menu` 与 `sys_role_menu` 的初始化方案。  
本次只输出设计文档，不实际执行 SQL。

## 2. 当前菜单体系说明

当前项目的菜单体系基于以下两张表：

- `sys_menu`
- `sys_role_menu`

其中：

- `sys_menu` 负责定义目录、页面菜单、按钮权限
- `sys_role_menu` 负责定义“角色拥有哪些菜单/按钮权限”
- 前端登录后根据当前用户角色加载菜单数据，再动态渲染左侧导航和页面按钮

这套机制已经具备企业后台的基本能力，因此知识库模块不需要重新发明一套权限导航机制，只需要按现有风格扩展即可。

## 3. 为什么要把知识库功能挂到 `sys_menu`

因为当前系统是“数据库驱动菜单”的模式，而不是前端写死菜单。

这样做的好处：

1. 不同角色看到的导航天然不同
2. 按钮权限可以和菜单权限统一管理
3. 后续新增功能时，只要插入菜单数据即可接入后台权限体系
4. 更适合企业项目里“可配置权限”的表达方式

## 7. 角色授权初始化建议

### 7.1 超级管理员

建议：

- 默认拥有知识库全部菜单和全部按钮权限

对应做法：

- 将 `menu_id` 100-161 全量分配给 `role_id = 1`

### 7.2 普通角色

建议默认只开放以下能力：

- 知识库工作台
- 文档管理
- 文档预览
- 文档下载
- 发起问答
- 查看自己的问答会话

不建议默认开放：

- 分类新增/删除
- 文档删除
- 任务重试
- AI审计导出

原因：

- 普通用户应该偏“使用者”
- 管理类能力应收敛给管理员或知识管理员

### 7.3 建议新增业务角色

为了让权限模型更像企业系统，建议后续新增以下角色：

1. `knowledge_admin`
2. `knowledge_editor`
3. `knowledge_user`
4. `ai_auditor`

角色建议说明：

- `knowledge_admin`：拥有知识库全部后台权限
- `knowledge_editor`：可上传、编辑、触发入库，但不能管理系统底座
- `knowledge_user`：可查看、预览、下载、问答
- `ai_auditor`：可查看 AI 审计，但不一定能改文档

## 9. 前端动态渲染导航是否算项目亮点

算，而且是可以写进简历和面试里的亮点。  
但建议不要只说“动态渲染导航”，而要说得更完整一些。

### 9.1 更推荐的表达方式

推荐写法：

- 基于 RBAC 的动态菜单与按钮权限控制
- 基于数据库菜单模型的前端动态路由渲染
- 基于角色授权的后台导航与操作权限隔离

### 9.2 为什么它算亮点

因为它体现了三个能力：

1. 前后端联动的权限设计能力
2. 菜单、路由、按钮统一建模能力
3. 企业后台场景下“不同角色看到不同系统能力”的实现能力

它不是一个简单的前端 UI 技巧，而是后台权限体系的一部分。

### 9.3 但它不应该是“唯一亮点”

如果只说“动态渲染导航”，亮点不够强。  
更好的讲法是把它放进一整套方案里：

- 用户、角色、部门、菜单、按钮权限统一建模
- 通过 `sys_menu + sys_role_menu` 控制页面和按钮权限
- 登录后根据角色动态返回菜单树并渲染导航
- 知识库模块新增菜单时无需写死前端导航

这样它就从“小技巧”变成了“权限系统设计亮点”。

## 10. 面试时怎么回答这个亮点

### 问：你项目里为什么要做动态菜单？

答：

因为这是一个企业后台系统，不同角色看到的功能范围本来就不一样。  
如果前端把菜单写死，不但扩展成本高，而且权限控制容易失真。  
所以我采用了 `sys_menu + sys_role_menu` 的建模方式，让菜单、路由和按钮权限都从数据库驱动。  
用户登录后，后端根据角色返回菜单树，前端再动态渲染导航和按钮，这样权限边界更清晰，也更接近真实企业系统。

### 问：这种设计的价值是什么？

答：

最大的价值是可配置、可扩展和角色隔离。  
新增一个知识库模块时，不需要改死前端导航结构，只要新增菜单数据并分配角色权限即可。  
这对于企业内部系统非常重要，因为权限变化会很频繁。

### 问：它和知识库业务有什么关系？

答：

知识库系统天然有很多角色差异。  
比如管理员可以建分类、删文档、重试入库；普通用户只能预览、下载和问答。  
动态菜单和按钮权限正好能把这种角色差异直接体现在系统入口和操作行为上。

## 11. 最终结论

知识库模块接入 `sys_menu` 的本质，不只是“多几个菜单”，而是把知识库功能完整纳入现有后台 RBAC 权限体系。

这样做的意义是：

1. 保持和现有系统架构一致
2. 支持按角色动态渲染导航
3. 支持按钮级权限控制
4. 更适合写进简历和面试，体现企业级后台设计能力
