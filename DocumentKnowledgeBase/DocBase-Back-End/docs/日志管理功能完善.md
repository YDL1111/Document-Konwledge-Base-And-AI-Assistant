# DocBase 日志管理功能完善

## 1. 文档目的

本文档用于指导完善系统管理模块中的“日志管理”功能，范围包括：

1. 操作日志
2. 登录日志

当前日志管理在页面上虽然存在，但实际业务价值不足，无法有效服务于当前“企业文档知识库系统”。  
本次改造目标是让日志管理从“模板占位功能”升级为“可用于追踪、审计、排查问题”的真实系统能力。

本文档将作为另一个开发 Agent 的执行依据。

---

## 2. 改造目标

本次日志管理完善要达到以下目标：

1. 登录日志能够真实记录用户登录、退出、登录结果等信息。
2. 操作日志能够真实记录系统管理与知识库管理中的关键业务行为。
3. 管理员可以通过日志管理页面进行查询、筛选、查看详情、必要的清理操作。
4. 日志内容能够用于排查权限问题、审核问题、上传下载问题、用户行为问题。
5. 日志能力要贴合当前项目，不要机械恢复与项目无关的模板逻辑。

---

## 3. 当前问题判断

目前日志管理模块主要存在以下问题：

1. 页面存在，但不一定有完整真实的数据支撑。
2. 操作日志对知识库模块的覆盖不足。
3. 登录日志可能没有完整记录成功/失败/退出等状态。
4. 日志查询能力不够完整，难以用于排查问题。
5. 日志详情信息不够清晰，难以体现企业项目中的审计能力。

---

## 4. 范围说明

本次只聚焦“系统管理 > 日志管理”：

1. 操作日志
2. 登录日志

本轮不要求做：

1. 服务器 CPU / 内存 / JVM 监控
2. 复杂链路追踪平台
3. 外部日志平台对接
4. ELK / Prometheus / Grafana 集成

---

## 5. 操作日志需要完善的功能

## 5.1 基础能力

必须具备：

1. 操作日志分页列表查询
2. 按操作人筛选
3. 按业务模块筛选
4. 按操作状态筛选
5. 按时间范围筛选
6. 查看日志详情
7. 清理操作日志

---

## 5.2 需要重点覆盖的业务模块

除了系统管理自身的日志外，必须重点补齐知识库相关操作日志。

至少应覆盖以下行为：

### 系统管理

1. 新增用户
2. 编辑用户
3. 重置密码
4. 角色新增/编辑/删除
5. 部门新增/编辑/删除
6. 岗位新增/编辑/删除
7. 公告新增/编辑/删除
8. 参数配置修改

### 知识库管理

1. 新增文档
2. 查看文档详情（可选，若日志粒度不宜过细可不记）
3. 审核通过文档
4. 审核驳回文档
5. 新增分类
6. 编辑分类
7. 删除分类
8. 下载文档
9. 预览文档（可选）
10. AI 问答关键调用（如当前已有审计需求）
11. AI 审计相关操作（如当前已有）

---

## 5.3 操作日志字段建议

列表页与详情页至少应能体现以下信息：

1. 日志 ID
2. 操作模块
3. 操作类型
4. 请求方法
5. 操作人用户名
6. 操作人 IP
7. 操作地址 / 登录地点（如已有）
8. 请求路径
9. 请求参数
10. 返回结果摘要
11. 操作状态（成功/失败）
12. 操作时间
13. 错误信息（若失败）

注意：

1. 日志详情中可以展示更完整的请求参数与异常信息。
2. 但列表页不应一次性展示过长文本，避免可读性差。

---

## 5.4 操作日志与当前项目的价值

这部分要能支持以下排查场景：

1. 谁上传了某个知识库文档
2. 谁审核通过/驳回了某个文档
3. 谁删除了某个分类
4. 谁修改了某个用户角色
5. 谁修改了系统参数
6. 某次失败操作的请求参数和错误原因是什么

---

## 6. 登录日志需要完善的功能

## 6.1 基础能力

必须具备：

1. 登录日志分页列表查询
2. 按用户名筛选
3. 按登录状态筛选
4. 按登录时间范围筛选
5. 查看登录日志详情（如当前 UI 有详情能力）
6. 清理登录日志

---

## 6.2 必须记录的登录行为

登录日志至少应记录：

1. 登录成功
2. 登录失败
3. 退出登录

如项目已有验证码、Token、Redis 登录态相关能力，也应确保其异常情况能反映到日志中。

---

## 6.3 登录日志字段建议

至少应体现：

1. 日志 ID
2. 用户名
3. 登录状态
4. 提示信息
5. IP 地址
6. 登录地点
7. 浏览器信息（如已有）
8. 操作系统信息（如已有）
9. 登录时间

---

## 6.4 登录日志与当前项目的价值

这部分要能支持以下场景：

1. 某个用户是否真的登录过系统
2. 登录失败是密码错误、账号禁用还是权限问题
3. 某个普通用户是否真的完成过登录并使用系统
4. 是否存在频繁失败登录的异常情况

---

## 7. 前端页面完善要求

日志管理页面不能只是展示表格，至少要具备以下交互能力：

### 操作日志页面

1. 查询表单可用
2. 重置表单可用
3. 查看详情弹窗可用
4. 清空日志按钮可用
5. 分页切换可用

### 登录日志页面

1. 查询表单可用
2. 重置表单可用
3. 清空日志按钮可用
4. 分页切换可用

页面层面要注意：

1. 去掉模板残留或“仅演示”式文案
2. 中文文案统一成当前 DocBase 项目风格
3. 列表字段与详情字段要和实际后端返回保持一致

---

## 8. 后端能力完善要求

另一个 Agent 执行时，需要重点检查并补齐：

1. 日志列表接口是否真实可查询
2. 查询条件是否完整生效
3. 分页是否正常
4. 清空接口是否真实生效
5. 日志详情接口是否可用
6. 系统中关键业务动作是否真的写入日志
7. 登录、退出、失败登录是否真的写入登录日志

---

## 9. 与知识库模块联动的重点要求

这是本次日志管理改造的核心亮点。

必须确保日志管理能够支撑知识库模块排查问题：

1. 文档上传失败时，能查到是谁发起的、请求是否成功、错误原因是什么。
2. 文档审核后，能查到是谁审核的、审核结果是什么。
3. 分类修改后，能查到是谁做了修改。
4. 权限问题排查时，能结合用户管理、角色管理、操作日志综合定位。
5. 普通用户看不到某个文档时，管理员可以通过日志辅助判断是权限问题、审核状态问题，还是根本未操作成功。

---

## 10. 建议开发顺序

建议另一个 Agent 按以下顺序执行：

1. 先核对后端日志表结构、日志写入链路、现有 controller/service
2. 修复登录日志的真实记录能力
3. 修复操作日志的真实记录能力
4. 优先补齐知识库关键业务的日志覆盖
5. 再补前端查询、详情、清空、分页等页面能力
6. 最后做联调回归

原因：

1. 先有真实数据，前端页面才有意义
2. 知识库相关日志是你当前项目最需要的部分

---

## 11. 禁止事项

1. 不允许只修前端页面，不修后端真实能力
2. 不允许日志表里有数据，但前端字段完全对不上
3. 不允许为了“看起来完整”而记录大量无意义噪音日志
4. 不允许破坏当前登录流程、知识库上传审核流程
5. 不允许把系统监控类功能混进本次日志管理改造范围

---

## 12. 最终验收标准

开发完成后，至少验证以下场景：

### 登录日志

1. 管理员登录后，能看到对应登录日志
2. 普通用户登录后，能看到对应登录日志
3. 登录失败后，能看到失败日志
4. 退出登录后，能看到退出日志
5. 登录日志可按用户名、状态、时间查询

### 操作日志

1. 新增用户后，操作日志可查
2. 编辑角色权限后，操作日志可查
3. 新增公告后，操作日志可查
4. 上传知识库文档后，操作日志可查
5. 审核通过/驳回知识库文档后，操作日志可查
6. 删除知识库分类后，操作日志可查
7. 操作日志可按模块、操作人、状态、时间筛选
8. 日志详情能看出关键信息

---

## 13. 交给开发 Agent 的执行要求

另一个 Agent 执行时，必须遵守以下要求：

1. 先分析现状，再改代码
2. 以“真实可用”为目标，而不是只把模板页面恢复出来
3. 每完成一块，都要说明：
   - 改了什么
   - 涉及哪些文件
   - 如何验证
   - 是否影响知识库模块
4. 操作日志与登录日志要分开梳理，但最终都要接入当前项目真实业务链路
5. 优先保证知识库关键日志可查，再考虑其他增强项
