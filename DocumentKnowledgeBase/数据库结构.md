# 数据库结构

当前项目对外统一使用数据库名：`docbase_knowledge`。

## 1. 现有基础表

基础管理表主要包括：

- `sys_user`
- `sys_role`
- `sys_menu`
- `sys_dept`
- `sys_post`
- `sys_config`
- `sys_notice`
- `sys_login_info`
- `sys_operation_log`
- `sys_role_menu`

这些表承担系统管理底座能力，不需要删除，而是继续作为企业文档知识库系统的基础设施复用。

## 2. 新增知识库核心表

为支撑企业文档知识库业务，新增设计如下表：

- `knowledge_category`
- `knowledge_document`
- `knowledge_document_version`
- `knowledge_document_acl`
- `knowledge_ingest_task`
- `ai_chat_session`
- `ai_chat_message`
- `ai_audit_log`

## 3. 设计原则

1. 保持与现有系统基础表相同的命名风格和审计字段风格。
2. 采用逻辑删除，便于后台系统统一管理。
3. 保持“业务表 + 审计字段 + 状态字段 + 索引字段”的一致风格。
4. 在不强依赖物理外键的前提下，通过应用层维护业务关联。

## 4. 面试问答建议

### 为什么不大量使用物理外键？

因为后台系统中很多表需要支持批量导入、逻辑删除、历史数据保留和跨模块演进。使用逻辑关联更利于扩展、迁移和性能控制。

### 为什么保留基础表不重建？

因为用户、角色、菜单、权限、日志这类能力本质上是企业后台的通用底座，复用成熟能力比全部重写更符合真实项目实践。

### 为什么知识文档要拆版本表？

因为企业文档天然存在版本迭代、回滚、解析重试、审计追踪需求，把文档主表和版本表拆开后更方便做状态管理和历史保留。
