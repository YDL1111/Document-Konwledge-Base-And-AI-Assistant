# 企业文档知识库与智能问答系统 —— 项目规划

> 最后更新：2026-05-12

## 架构总览

```
Java 端（业务底座） — 文档管理 / 权限 / 审计 / 前端交互
         ↕ 内部 REST API（异步任务 + API Key 鉴权）
Python 端（AI 大脑） — 文档解析 / 向量化 / 混合检索 / 大模型问答
```

---

## ☕ Java 项目：企业文档与知识管理平台（业务底座）

**定位：稳。** 做好业务逻辑、权限控制、文档全生命周期管理。

### 1. 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| 核心框架 | Spring Boot 3.x | 初期单体即可，后续需微服务再引入 Spring Cloud Alibaba |
| 持久层与数据库 | MyBatis-Plus + MySQL 8.0 | 文档元数据、用户、权限、审计日志 |
| 缓存与权限 | Redis + Sa-Token | Redis 做热点缓存/分布式锁；Sa-Token 做 RBAC（比 Spring Security 轻量） |
| 全文检索 | Elasticsearch 8.x | 文档标题、标签、元数据的传统检索，不碰向量检索 |
| 文件存储 | MinIO | 私有化对象存储，S3 协议兼容 |
| 消息队列 | RabbitMQ 或 Redis Stream | Java → Python 的异步任务通知，解耦 + 削峰 |
| 前端 | Vue 3 + Element Plus + TypeScript | |
| 文档预览 | kkFileView | 支持 PDF / Word / Excel / PPT 在线预览 |
| 容器化 | Docker + Docker Compose | 本地开发 & 演示一键启动 |

### 2. 核心功能模块

#### 2.1 多格式文档管理

- 支持 PDF / Word / Excel / PPT / Markdown / 纯文本上传
- 文件上传后存入 MinIO，MySQL 记录元数据（标题、作者、部门、标签）
- 在线预览（kkFileView），版本控制（保留历史版本，支持回滚）
- ⚠ **关键：Java 端不做文档解析！** 只负责上传存储和通知 Python 端处理

#### 2.2 细粒度权限体系

- 部门 / 角色 / 用户 三级 RBAC 权限
- 文档级别权限：私有 / 部门可见 / 全员公开
- 权限数据随请求传递给 Python 端，用于检索过滤
- 示例：研发部用户检索时只命中研发文档 + 公开文档

#### 2.3 文档生命周期管控

- 起草 → 审核（Flowable 工作流） → 发布 → 归档 → 下架
- 文档更新时自动触发向量库增量更新（先删旧 chunk 再索引新 chunk）
- 每个 chunk 关联 `document_id` + `version_id`，支持按版本检索

#### 2.4 操作审计日志

- 记录：谁 / 什么时间 / 对哪个文档 / 做了什么操作
- 操作类型：上传、下载、预览、修改、删除、检索
- 支持按用户 / 时间 / 文档维度查询

#### 2.5 AI 交互代理层

- 前端不直接调 Python，所有 AI 请求经 Java 中转
- Java 负责：用户鉴权 → 附加权限上下文 → 转发 Python → 结果渲染
- 前端看到：答案 + 溯源文档链接（可跳转预览） + 引用段落高亮

## 🐍 Python 项目：RAG 智能问答助手（AI 大脑）

**定位：智。** 搞定非结构化数据解析、向量化、混合检索、大模型调度。

### 1. 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| Web 框架 | FastAPI | 异步高性能，天然适合 AI 推理服务 |
| AI 编排框架 | LlamaIndex | 比 LangChain 更专注于 RAG，API 更简洁 |
| 向量数据库 | Milvus 2.x | 企业级，支持 metadata 过滤；备选 Qdrant（更轻量） |
| 大模型（生成） | **DeepSeek v4 Flash** | 云端 API，速度快、成本低、中文能力强，兼容 OpenAI 格式 |
| Embedding 模型 | BGE-M3 | 智源出品，支持中文/多语言，1024 维向量 |
| Rerank 模型 | BGE-Reranker-v2-M3 | 对混合检索召回结果做精准打分排序 |
| 文档解析 | PyMuPDF + python-docx + openpyxl | PDF / Word / Excel；备选 Unstructured（统一解析多格式） |
| 异步任务 | Celery + Redis 或 asyncio BackgroundTasks | 文档入库异步处理，避免 HTTP 超时 |

### 2. 核心功能模块

#### 2.1 智能文档解析与切分

**解析阶段：**

- 从 MinIO 拉取文件（通过 MinIO SDK，不经过 Java）
- **PDF**：PyMuPDF 提取文本 + 页面元数据（页码用于溯源）
- **Word**：python-docx 提取正文，过滤页眉页脚和批注
- **Excel**：openpyxl 保留行列关系，转为 Markdown 表格或结构化 JSON
- **去噪**：移除乱码、多余空白、重复内容

**切分策略（分层处理，不用固定 500 token 一刀切）：**

- 第一层：按语义边界切（标题层级、段落）
- 第二层：超过阈值再按大小切（`RecursiveCharacterTextSplitter`）
- 参数建议：`chunk_size=500~800 字`（中文字），`overlap=50~100 字`
- 特殊结构：表格单独作为一个 chunk，不拆分
- 每个 chunk 携带 metadata：`{ document_id, title, version, department, page, chunk_index }`

#### 2.2 向量化与存储

调用 BGE-M3 将每个 chunk 转为向量，存入 Milvus：

| 字段 | 类型 | 用途 | 备注 |
|------|------|------|------|
| `chunk_id` | VARCHAR | 唯一标识 | UUID |
| `document_id` | VARCHAR | 文档关联 | 外键关联 |
| `title` | VARCHAR | 文档标题 | 展示用 |
| `content` | VARCHAR | chunk 文本 | 溯源用 |
| `embedding` | FLOAT[] | 向量 | 1024 维 |
| `department` | VARCHAR | 部门标签 | 权限过滤 |
| `visibility` | VARCHAR | 可见范围 | private / dept / public |
| `version` | VARCHAR | 版本号 | 版本管理 |
| `page` | INT | 页码 | 溯源定位 |

> 文档更新流程：先 `delete` `document_id`+`version` → 再 `insert` 新 chunk

#### 2.3 混合检索与重排

**检索流程：**

```
用户问题 → 并行两路召回 → Rerank 排序 → 取 top-K 返回
```

**第一路 · 向量语义检索（Milvus）：**
- 问题 → BGE-M3 → 向量 → Milvus 相似性搜索
- 带 metadata 过滤：`department` / `visibility` 权限控制
- 召回 top-20

**第二路 · 关键词检索（Elasticsearch，可选）：**
- 问题分词 → ES 匹配 chunk content / title
- 适用于专有名词、数字、精确匹配场景
- 召回 top-10

**合并与重排：**
- 两路结果去重合并（最多 30 个候选）
- BGE-Reranker 对候选 chunk 重新打分排序
- 取最终 top-5 作为 RAG 上下文

**调参建议：**
- 初期先跑通向量单路检索，再加关键词 + Rerank
- Rerank 阈值：低于 `0.3` 的分数直接丢弃，返回"未找到相关信息"

#### 2.4 RAG 答案生成与安全护栏

**Prompt 模板：**

```
你是一个企业知识问答助手。请仅根据以下参考材料回答问题。

【参考材料】
{context}

【用户问题】
{question}

要求：
1. 仅根据参考材料回答，不要使用你的训练数据
2. 如果参考材料中找不到相关信息，直接回答"抱歉，当前知识库中没有
   找到与您的问题相关的内容"
3. 回答时标注信息来源，格式为：[来源：《文档标题》第X页]
4. 回答简洁准确，不要过度展开
5. 涉及数据、日期、人名时，严格使用材料中的原始值
```

**安全护栏：**
- 输入过滤：拦截注入攻击（如"忽略以上指令"类 prompt injection）
- 输出校验：检查是否包含 `[来源：...]` 标注
- 置信度控制：Rerank 最高分 < 阈值时，不生成答案，直接告知无结果

## 🔗 两个项目的联动设计

### 1. 接口定义（Java ↔ Python）

> 通信方式：HTTP REST API（内网通信，API Key 鉴权）
> 任务通知：通过消息队列（RabbitMQ / Redis Stream）异步触发解析

#### 接口 1：文档入库（异步）

```http

Content-Type: application/json

  "visibility": "public",

}
```

### 2. 关键交互流程

#### 知识入库流程

```
用户上传文件
    │
    ▼
Java: 文件存入 MinIO → MySQL 记录元数据
    │
    ▼
Java: 发送消息到队列（或 HTTP 调 /ingest）
    │
    ▼
Python: 从队列消费任务
    │
    ▼
Python: 从 MinIO 下载文件 → 解析 → 切分 → 向量化
    │
    ▼
Python: 存入 Milvus → 更新任务状态
    │
    ▼
Java: 轮询 / WebSocket 通知前端 → 显示"解析完成"
```

#### 智能问答流程

```
用户输入问题
    │
    ▼
前端 → Java: 鉴权 + 附加权限上下文 → Python /query
    │
    ▼
Python: 问题向量化 → Milvus 检索（带权限过滤）
    │
    ▼
Python: （可选）ES 关键词检索 → 合并去重
    │
    ▼
Python: Rerank 排序 → 取 top-5 作为上下文
    │
    ▼
Python: 组装 Prompt → 调用 DeepSeek v4 Flash API → 生成答案
    │
    ▼
Python: 返回 { answer, sources[] } → Java → 前端渲染
    │
    ▼
前端: 展示答案 + 溯源链接（点击跳转文档预览对应页码）
```

### 3. 权限传递方案

Java 端在调用 Python 的 `/query` 接口时，附带用户权限信息：

```json
{
  "user_id": "user_zhangsan",
  "department": "hr",
  "visibility_level": ["private", "hr", "public"]
}
```

Python 端在 Milvus 检索时转换为过滤表达式：

```python
filter = '(department == "hr" OR visibility == "public") AND visibility != "restricted"'
```

确保用户只能检索到自己有权限的文档 chunk。

### 4. 内部通信安全

- Java ↔ Python 走内网，不暴露公网
- 每个请求头带 API Key：`X-API-Key: <shared_secret>`
- Python 端中间件验证 API Key，不匹配直接 403
- 可选：请求体 HMAC 签名，防篡改

## ⚠️ 避坑指南

### Java 端

1. **不要自己做 PDF / Word 解析！** 交给 Python
2. **不要自己做向量化！** Java 的向量生态薄弱
3. Elasticsearch 只用于传统元数据检索，别跟向量检索搞混
4. 微服务初期别上！单体 Spring Boot 足够，复杂度够呛了
5. 文档更新时一定要同步清理旧 chunk，否则向量库里会有脏数据

### Python 端

1. **别一上来就上混合检索 + Rerank！** 先单路向量检索跑通
2. Embedding 模型别用英文模型（如 OpenAI text-embedding-3），中文效果差，用 BGE-M3
3. Chunking 别固定 500 token！按语义边界切，中文按字算
4. **异步处理一定要做！** 大 PDF 解析要几分钟，同步 HTTP 必超时
5. Prompt 里必须约束"找不到就说不找"，否则模型会编

### 联调

1. MinIO 内部网络地址要 Java 和 Python 都能访问
2. 日志一定要对齐！`task_id` 贯穿全流程，方便排查问题
3. 先本地 docker-compose 跑通，再考虑 K8s

### 模型调用

1. DeepSeek v4 Flash API 做好限流和重试（指数退避）
2. Embedding 模型如果本地部署，注意 GPU 显存占用
3. 记录每次 API 调用的 token 消耗，方便估算成本

## 📌 关键设计决策记录

**Q: 为什么选 LlamaIndex 而不是 LangChain？**

LlamaIndex 更专注于数据索引和 RAG 场景，API 更简洁。LangChain 更通用但更重，适合 Agent 场景。后续做 Agent 时可以引入 LangGraph。

**Q: 为什么 Java 端不做文档解析？**

Java 的 PDF/Word 解析生态弱（Tika 效果一般），Python 的 PyMuPDF / Unstructured 生态成熟，效果好得多。

**Q: 向量数据库选 Milvus 还是 Qdrant？**

Milvus 功能全（企业级），Qdrant 更轻量。如果团队对运维有要求选 Milvus，追求简单选 Qdrant。初期 Qdrant 够用了。

**Q: 为什么用异步任务而不是同步调用？**

大文档解析 + 向量化可能耗时数分钟，同步 HTTP 调用必然超时。异步 + 状态轮询是标准做法。

**Q: DeepSeek v4 Flash 的优势？**

速度快、成本低、中文能力强，适合企业 RAG 场景。API 兼容 OpenAI 格式，接入无门槛。
