# 知识库SQL设计

本文档对应 `DocBase 企业文档知识库系统` 的最终 SQL 设计思路，当前实际数据库名称统一为 `docbase_knowledge`。

## 1. 设计目标

1. 复用后台基础表，不重复造轮子。
2. 为知识分类、知识文档、版本、权限、AI 问答设计独立业务表。
3. 在保证落地效率的同时，兼顾扩展性和面试可讲述性。

## 2. 核心业务表

### `knowledge_category`

用于维护知识分类树，支持多级目录、排序、状态控制。

### `knowledge_document`

用于维护文档主记录，保存标题、分类、标签、可见范围、当前版本等信息。

### `knowledge_document_version`

用于保存每次上传或变更后的版本快照，支持预览、回滚、解析状态跟踪。

### `knowledge_document_acl`

用于维护文档级访问控制，可按用户、角色、部门设置权限。

### `knowledge_ingest_task`

用于记录文档解析、切片、向量化、索引构建等异步任务状态。

### `ai_chat_session`

用于记录 AI 问答会话。

### `ai_chat_message`

用于记录 AI 问答消息明细。

### `ai_audit_log`

用于记录 AI 调用、命中来源、耗时、状态、错误信息等审计数据。

## 3. 索引设计原则

1. 优先围绕查询条件建立组合索引。
2. 对状态字段、时间字段、所属部门字段建立常用索引。
3. 对文档标题等高频搜索字段建立前缀索引。
4. 对版本号、任务号、唯一编码类字段建立唯一索引。

## 4. 面试问答建议

### 为什么 ACL 单独建表？

因为文档权限模型可能同时面向用户、角色、部门，如果直接塞进主表会导致字段膨胀，也不利于后续扩展更复杂的授权策略。

### 为什么导入任务单独建表？

因为知识库导入链路通常是异步流程，独立任务表更方便记录重试次数、错误信息、耗时、追踪 ID 和任务状态。

### 为什么 AI 审计日志要单独保留？

因为企业场景下需要解释模型调用、来源依据、结果状态和成本消耗，审计日志是系统合规性与可追溯性的关键支撑。
